# 简历优化 Agent — 需求分析与任务说明

本文档为「可运行的简历优化 Agent 应用」的**需求分析**、**任务描述**与**分阶段任务步骤**，与题目中的运行与交付要求对齐，作为后续开发与验收的参考依据。

---

## 1. 项目概述

### 1.1 背景与问题

求职者常常不清楚如何针对目标岗位调整简历：哪些经历应突出、哪些能力与岗位描述（JD）不匹配、如何改写更易通过筛选。需要一套工具化流程，将「简历 + 目标 JD」转化为可理解的匹配诊断与可落地的改写结果。

### 1.2 项目目标

构建一个**可运行的简历优化 Agent 应用**，支持用户上传或输入简历内容、输入目标职位描述，由系统：

1. 分析简历与 JD 的匹配情况，并结构化输出：**匹配亮点**、**主要缺口**、**具体优化建议**；
2. 在用户**明确确认**后，生成一份**优化后的完整简历内容**；
3. 支持用户**查看**与**导出**优化结果。

### 1.3 技术与安全约束

- **大模型服务**：题目指定使用 **DeepSeek** API（兼容 OpenAI 风格的调用方式可在实现阶段确定）。
- **API Key**：**必须通过环境变量配置**，禁止硬编码于代码、镜像或仓库中；文档与示例中不得写入真实密钥。
- **建议环境变量命名**（实现阶段可微调并在 README 中统一说明）：
  - `DEEPSEEK_API_KEY`：访问 DeepSeek 的密钥；
  - 可选 `DEEPSEEK_BASE_URL`：若需自定义 API 基础地址时使用。

## 2. 需求分析

### 2.1 用户与使用场景

- **用户**：正在求职、需要针对特定岗位优化简历的人员。
- **典型流程**：准备简历文本 → 获取或粘贴目标岗位 JD → 查看系统给出的匹配分析与建议 → 确认后获得优化版简历 → 复制或导出用于投递。

### 2.2 功能需求（必选）


| 编号  | 需求项     | 说明                                                                                                                             |
| --- | ------- | ------------------------------------------------------------------------------------------------------------------------------ |
| F1  | 简历输入    | 支持**上传文件**（PDF、Word **.docx**、TXT、Markdown 等）和/或**文本框直接输入**；服务端将文件转为纯文本并做基础去噪后再送模型；旧版 **.doc** 需用户先另存为 .docx 或 PDF；对空内容做校验与提示。 |
| F2  | JD 输入   | 支持用户输入或粘贴**目标职位描述**全文。                                                                                                         |
| F3  | 匹配分析    | 基于简历与 JD，输出三部分：**匹配亮点**、**主要缺口**、**具体优化建议**（建议为分点、可执行）。                                                                        |
| F4  | 确认后再生成  | 展示分析结果后，用户需**显式确认**（如按钮或勾选），系统才调用模型生成**优化后的完整简历正文**。                                                                           |
| F5  | 结果展示与导出 | 展示优化结果；支持复制与导出（docx / pdf / txt / md）。                                                                                 |
| F6  | 模板改写与预览 | 从模板仓库选择 LaTeX 模板，将“优化稿+模板内容”交给模型生成；展示整份预览图，支持反馈重写后下载 ZIP（含 tex/资源/pdf）。 |


### 2.3 非功能需求

- **可部署性**：应用须能通过 **Docker** 构建与启动；仓库根目录必须包含 **Dockerfile**；可按需提供 **docker-compose.yml**。
- **可交付文档**：必须提供完整 **README.md**，使面试官**仅根据 README** 即可完成环境配置、构建、启动与基本验收。
- **可维护性**：与 LLM 的交互（Prompt、解析、错误处理）结构清晰，便于后续调整模型或提示词。
- **输出质量**：分析与优化结果应对求职场景有**实际参考价值**（避免空泛套话，建议贴近 JD 中的能力项与表述）。

### 2.4 交付物清单

- 根目录 **Dockerfile**（必选）。
- 可选 **docker-compose.yml**。
- 完整 **README.md**（环境变量、构建镜像、启动服务、访问方式、使用步骤）。
- **可公开访问的 GitHub 仓库**（由项目维护者创建与推送，不在本文档中绑定具体 URL）。

## 3. 功能规格与验收要点

### 3.1 输入规格

- **简历**：文件上传（如纯文本、Markdown 等，支持范围在实现阶段定义）与/或多行文本输入；需处理编码与过大文件的基本限制。
- **JD**：自由文本；可将「附录：示例 JD」作为**手动测试用例**，**不应**将示例 JD 作为生产环境默认值硬编码进业务逻辑。

### 3.2 分析输出（结构化）

系统应对每次「分析」请求给出清晰可区分的三块内容：

1. **匹配亮点**：与 JD 强相关的经历、项目、技能或成果（宜引用简历中的可对应表述）。
2. **主要缺口**：相对 JD 的不足、未覆盖的关键词或薄弱项。
3. **具体优化建议**：可执行的改写方向（例如补充哪些关键词、如何重排模块、哪些 bullet 需量化等），以分点列出。

### 3.3 确认与生成

- 用户阅读分析结果后，通过界面**明确操作**表示同意继续（例如「生成优化简历」按钮）。
- 仅在确认后发起第二次 LLM 调用，生成**完整、可直接使用或略作修改的优化后简历正文**（而非仅片段补丁）。

### 3.4 结果与导出

- 页面清晰展示优化后的简历全文。
- 提供复制与文件导出能力；导出文件名与编码在 README 中说明。
- 模板改写场景下，需先生成预览图，再允许用户反馈重写与最终下载 ZIP。

### 3.5 模板改写返修记忆（新增）

- 同一模板的连续返修应具备上下文记忆（上一轮结果作为参考），降低重复报错与风格漂移。
- 用户切换模板时必须清空上一模板上下文，防止跨模板污染。

## 4. 与 LLM 的交互设计（需求层）

### 4.1 配置方式

- 所有密钥与敏感配置仅通过**环境变量**（或运行时注入，如 Docker `--env-file`）提供。
- 不在日志、前端网络面板说明或错误堆栈中回显完整 API Key 与用户隐私全文简历（实现阶段按最小必要原则处理）。

### 4.2 建议的两阶段调用

为便于控制成本、测试与产品逻辑拆分，建议分为两次主要调用：

1. **分析阶段**
  - **输入**：原始简历文本 + JD 文本。  
  - **输出**：结构化内容：匹配亮点、主要缺口、具体优化建议（可采用 JSON 约定或严格分节文本，由实现阶段定义解析方式）。
2. **优化阶段**（仅用户确认后）
  - **输入**：原始简历 + JD + 分析阶段摘要（或完整分析结果，以 Prompt 设计为准）。  
  - **输出**：优化后的**完整简历正文**。

### 4.3 工程原则

- Prompt 模板与业务路由、解析逻辑**分离**，便于迭代。
- 网络超时、限流与可理解的错误提示（避免向用户暴露内部堆栈）。
- 不对同一操作在无确认情况下重复自动调用高成本生成接口。

## 5. 任务描述（开发任务书）

实现一个**可运行的 Web 应用**（具体技术栈在实现阶段选定，例如 Python FastAPI + 轻量前端，或一体化全栈框架），提供简历与 JD 的录入界面，在服务端通过环境变量配置的 **DeepSeek API** 完成「匹配分析」与「确认后的简历优化生成」，使用 **Docker** 封装运行环境，并通过 **README.md** 说明环境变量、镜像构建、服务启动与访问地址，使面试官能够独立完成部署与基本功能验收。

## 6. 任务步骤（分阶段）

### 阶段 A — 文档与仓库骨架

1. 维护本文档 **R_A.md**，并与实现保持一致迭代。
2. 初始化应用目录结构、依赖声明（如 `requirements.txt`、`package.json` 等，依技术选型而定）。
3. 提供 `**.env.example`**（或等价说明），仅含占位符与注释，**不包含**真实密钥。

### 阶段 B — 核心功能

1. 实现简历与 JD 的前端输入与后端接收接口。
2. 实现「分析」流程：调用 LLM、解析结构化输出、返回给前端展示。
3. 实现「用户确认」门禁与「优化简历」生成流程。
4. 实现模板改写流程：模板选择、改写进度、整份预览、反馈重写、ZIP 导出。
5. 实现优化结果的展示、复制与导出。

### 阶段 C — 容器与文档

1. 编写根目录 **Dockerfile**（可选用多阶段构建以减小镜像体积）。
2. 按需编写 **docker-compose.yml**（端口映射、环境变量注入等）。
3. 编写 **README.md**：环境变量说明、构建命令、启动命令、访问 URL、基本验收步骤（可含截图占位说明）。

### 阶段 D — 质量与交付

1. 在本地与 Docker 环境下进行冒烟测试；全仓库检索确认**无 API Key 硬编码**。
2. 将代码推送到**可公开的 GitHub 仓库**（由维护者执行）。

### 阶段 E — 风险与关注点

- **空简历 / 空 JD**：前端与后端双重校验与友好提示。
- **API 超时与失败**：重试策略、降级提示（不泄露密钥）。
- **模板返修**：同模板记忆与跨模板隔离都需通过测试用例覆盖。
- **导出文件**：编码（UTF-8）与换行符在 Windows/Mac 下的可读性。
- **验收**：面试官仅依赖 README 即可完成构建、启动与一轮完整业务流程验证。
